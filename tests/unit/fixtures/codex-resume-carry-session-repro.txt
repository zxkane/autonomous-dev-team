Investigation artifact (#198): does `codex exec resume <tid>` carry session on
amazon-bedrock?  ANSWER: YES (codex-cli 0.137.0, model_provider=amazon-bedrock,
openai.gpt-5.5, region us-east-2). Captured 2026-06-07 on the dev box.

Repro (the exact `remember 42` test from the issue body):

  export CODEX_CMD="$(ls ~/.nvm/versions/node/*/bin/codex | head -1)"
  export AWS_REGION=us-east-2          # model openai.gpt-5.5 lives here
  TID=$("$CODEX_CMD" exec --json "Remember the number 42." </dev/null \
          | grep -m1 -o '"thread_id":"[^"]*"' | cut -d'"' -f4)
  "$CODEX_CMD" exec resume "$TID" --json "What number did I ask you to remember?" </dev/null \
          | grep -E '"type":"thread.started"|"type":"agent_message"'

Observed (turn-1 thread_id = 019ea217-f3d5-72e2-9eb0-7b7a1544436f):

  turn-1 (exec, --json):
    {"type":"thread.started","thread_id":"019ea217-f3d5-72e2-9eb0-7b7a1544436f"}
    agent_message: "OK 42"
    turn.completed usage: input_tokens=17799 output_tokens>0

  resume-1 (exec resume <tid>):
    agent_message (--output-last-message): "RECALL 42"   <-- recalled prior turn

  resume-2 (exec resume <tid> --json):
    {"type":"thread.started","thread_id":"019ea217-f3d5-72e2-9eb0-7b7a1544436f"}  <-- SAME id
    {"type":"agent_message", ... "AGAIN 42"}
    turn.completed usage:
      input_tokens=53500 cached_input_tokens=46817 output_tokens=37

Conclusions:
  1. Resume KEEPS the SAME thread.started id across turns (the issue's RC1
     predicted a NEW id every resume — that prediction does NOT hold on 0.137.0).
  2. input_tokens grows turn-over-turn with cached_input_tokens populated — the
     prior conversation is replayed and Bedrock-prompt-cached, NOT re-read from
     zero.
  3. Recall works (OK 42 -> RECALL 42 -> AGAIN 42).

Therefore RC1 ("resume is a no-op on amazon-bedrock") is NOT reproducible on the
current CLI; the resume loop is sound and is KEPT. The real defect is RC2 — the
convergence detector false-positives on progress narration (see
codex-gather-only-turn.jsonl) — which this PR fixes.

NOTE on the harmless `app_session_terminated` / "Failed to refresh token" 400
banners that appear on stderr during resume: these are the stale ChatGPT-login
refresh noise; the bedrock model still answers rc 0 (confirmed by RECALL 42 /
AGAIN 42 above). They are NOT the cause of the drop. (A genuinely expired
~/.codex/auth.json is a separate, interactive-fix failure mode — distinguishable
by codex exiting ~30s with NO model output at all.)
